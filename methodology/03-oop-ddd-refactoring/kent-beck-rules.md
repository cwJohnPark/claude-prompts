# Kent Beck의 TDD 및 Tidy First 원칙

Kent Beck의 Test-Driven Development (TDD)와 Tidy First 원칙을 따르는 개발 가이드입니다.

## 핵심 개발 원칙

- 항상 TDD 사이클을 따른다: Red → Green → Refactor
- 가장 단순한 실패 테스트를 먼저 작성한다
- 테스트를 통과시키는 최소한의 코드만 구현한다
- 테스트가 통과한 후에만 리팩토링한다
- Beck의 "Tidy First" 접근법에 따라 구조적 변경과 행위적 변경을 분리한다
- 개발 전체에 걸쳐 높은 코드 품질을 유지한다

## TDD 방법론 가이드

- 작은 기능 증분을 정의하는 실패 테스트 작성으로 시작한다
- 동작을 설명하는 의미 있는 테스트 이름을 사용한다 (예: "shouldSumTwoPositiveNumbers")
- 테스트 실패를 명확하고 정보가 풍부하게 만든다
- 테스트를 통과시키는 코드만 작성한다 — 그 이상은 안 된다
- 테스트 통과 후, 리팩토링이 필요한지 고려한다
- 새로운 기능을 위해 사이클을 반복한다
- 결함 수정 시: 먼저 API 수준의 실패 테스트를 작성하고, 문제를 재현하는 가장 작은 테스트를 작성한 후, 두 테스트를 모두 통과시킨다

## Tidy First 접근법

- 모든 변경을 두 가지 유형으로 분리한다:
    1. STRUCTURAL CHANGES: 동작을 변경하지 않고 코드 구조만 개선 (이름 변경, 메서드 추출, 코드 이동)
    2. BEHAVIORAL CHANGES: 실제 기능 추가 또는 수정
- 구조적 변경과 행위적 변경을 동일 커밋에 혼합하지 않는다
- 두 가지 모두 필요할 때는 항상 구조적 변경을 먼저 수행한다
- 테스트를 전후에 실행하여 구조적 변경이 동작을 바꾸지 않았음을 검증한다

## 커밋 규율

- 커밋 조건:
    1. 모든 테스트가 통과
    2. 모든 컴파일러/린터 경고가 해결됨
    3. 변경이 단일 논리 단위를 나타냄
    4. 커밋 메시지에 구조적/행위적 변경 여부를 명시
- 크고 드문 커밋보다 작고 빈번한 커밋을 사용한다

## 코드 품질 기준

- 중복을 철저히 제거한다
- 이름과 구조를 통해 의도를 명확히 표현한다
- 의존성을 명시적으로 만든다
- 메서드를 작게 유지하고 단일 책임에 집중한다
- 상태와 부수 효과를 최소화한다
- 동작 가능한 가장 단순한 해결책을 사용한다

## 리팩토링 가이드라인

- 테스트가 통과하는 상태("Green" 단계)에서만 리팩토링한다
- 정립된 리팩토링 패턴을 올바른 이름으로 사용한다
- 한 번에 하나의 리팩토링만 수행한다
- 각 리팩토링 단계 후 테스트를 실행한다
- 중복 제거 또는 명확성 개선 리팩토링을 우선시한다

## 워크플로우 예시

새 기능 접근 시:

1. 기능의 작은 부분에 대한 단순한 실패 테스트 작성
2. 통과시키기 위한 최소한의 구현
3. 테스트 실행으로 통과 확인 (Green)
4. 필요한 구조적 변경 수행 (Tidy First), 각 변경 후 테스트 실행
5. 구조적 변경을 별도 커밋
6. 다음 작은 기능 증분을 위한 테스트 추가
7. 기능 완성까지 반복, 행위적 변경을 구조적 변경과 별도 커밋

이 프로세스를 정확히 따르되, 항상 깨끗하고 잘 테스트된 코드를 빠른 구현보다 우선시한다.

항상 한 번에 하나의 테스트만 작성하고, 실행하고, 구조를 개선한다. 매번 모든 테스트를 실행한다.
