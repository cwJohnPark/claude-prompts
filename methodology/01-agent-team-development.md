# Agent-Team 중심 개발 가이드

LLM 코딩 시 흔히 발생하는 실수를 줄이기 위한 행동 지침.
프로젝트별 지침과 병합하여 사용한다.

**기본 방침:** 속도보다 신중함을 우선한다. 단순한 작업에는 재량껏 판단한다.

---

## 1. Agent-Team 구성 및 역할

오케스트레이터가 아래 에이전트를 순서대로 spawn한다.

| 에이전트 | 담당 Phase | 핵심 책임 | 금지 사항 |
|---------|-----------|----------|----------|
| **API Researcher** | Phase 1 — 설계 | 소스 탐색, 도메인 설계, 인수 조건 작성 | 코드 작성, 파일 수정 |
| **Developer** | Phase 2 — 개발 | 프로덕션 코드 구현 | 테스트 작성 |
| **Test Writer** | Phase 2 — 개발 (병렬) | BDD 테스트 작성 | Mock 라이브러리 사용 |
| **Refactoring Specialist** | Phase 3 — 리팩토링 | 객체지향 체조 엄격 적용 | Phase 2에서 변경한 파일 외 수정 |
| **Validator** | Phase 4 — 검증 | 테스트 실행 + E2E 검증 | 코드 수정 |

---

## 2. 무한 루프 — 4-Phase 개발 사이클

```
루프 시작: 사용자가 "다음 기능" 또는 "루프 시작"을 요청하면 Phase 1부터 시작.
```

### Phase 1 — 설계 (Design) · API Researcher

- ROADMAP에서 첫 번째 미완료 항목을 선택한다.
- 관련 소스를 탐색하고, 구현 계획을 작성한다 (변경 파일 목록, 인수 조건 Given/When/Then).
- **게이트:** 사용자 승인 없이 Phase 2 진행 금지.

### Phase 2 — 개발 (Development) · Developer + Test Writer (병렬)

BDD 방식의 TDD를 따른다.

- **테스트 프레임워크:** 프로젝트에 적합한 BDD 프레임워크 사용
- **목 라이브러리 금지:** 실제 객체, 인메모리 DB, 수동 Fake만 허용
- **테스트 3계층:**
  - 계층 1 — 순수 도메인: 프레임워크 없음, 직접 객체 생성
  - 계층 2 — 통합: 프레임워크 테스트 + 인메모리 DB
  - 계층 3 — 외부 API: 수동 Fake 구현체
- **절차:** 스펙 작성(RED) → 실패 확인 → 구현(GREEN) → 통과 확인
- **게이트:** 전체 테스트 통과 필수.

### Phase 3 — 리팩토링 (Refactoring) · Refactoring Specialist

변경한 파일에 대해서만 9가지 객체지향 체조 규칙 체크리스트를 적용한다.
각 리팩토링 후 테스트 통과를 확인한다.

- **게이트:** 전체 테스트 통과 필수.

### Phase 4 — 검증 (Verification) · Validator

- **전제:** 앱이 실행 중이어야 한다.
- 앱 탐색 → 스크린샷 → 상호작용 → 보고서 생성
- 문제 발견 시 Phase 2로 복귀
- **게이트:** E2E 검증 완료 후 사용자에게 결과 보고.

### 루프 완료

- 사용자에게 커밋을 제안한다.
- ROADMAP의 완료 항목을 체크한다.
- 사용자가 "계속"이라고 하면 Phase 1로 복귀한다.

## 3. Refactoring Specialist 엄격 규칙 (예외 없음)

위반 발견 시 즉시 수정한다.

| # | 규칙 | 수정 방향 |
|---|------|----------|
| 1 | `for`/`forEach` 금지 | 함수형 API 사용 (`map`, `filter`, `firstOrNull`, `flatMap`, `fold`) |
| 2 | `else`/`else if` 금지 | early return 또는 `when` 사용 |
| 3 | 원시값으로 도메인 표현 금지 | 값 객체(Value Object)로 포장 |
| 4 | 한 줄에 점(`.`) 2개 이상 금지 | 줄바꿈으로 분리 |
| 5 | 축약 이름 금지 | 완전한 이름 사용 |
| 6 | 파일 100줄 초과 금지 | 역할별로 분리 |
| 7 | 생성자 파라미터 4개 이상 금지 | 값 객체로 병합 |
| 8 | `List<도메인>` 직접 노출 금지 | 일급 컬렉션으로 감싸기 |
| 9 | 외부에서 프로퍼티 꺼내 판단 금지 | 객체 내부 메서드로 이동 |

## 4. 테스트 인프라 규칙

- **BDD 프레임워크:** 프로젝트에 적합한 BDD 프레임워크 사용
- **금지:** Mock 라이브러리 사용
- **허용:** 수동 Fake, 테스트 설정, 테스트 전용 Bean

---

## 5. 코딩 전 원칙 — 먼저 생각하라

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현 전에 반드시:
- 가정을 명시적으로 진술한다. 불확실하면 질문한다.
- 해석이 여러 가지라면 모두 제시한다 — 조용히 하나만 고르지 않는다.
- 더 단순한 접근이 있으면 말한다. 타당할 때는 반론을 제기한다.
- 무언가 불명확하면 멈춘다. 무엇이 혼란스러운지 명명하고 질문한다.

---

## 6. 단순함 우선

**문제를 해결하는 최소한의 코드만 작성한다. 추측성 코드는 없다.**

- 요청받지 않은 기능을 추가하지 않는다.
- 한 번만 쓰이는 코드에 추상화를 만들지 않는다.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 넣지 않는다.
- 불가능한 시나리오에 대한 에러 처리를 하지 않는다.
- 200줄로 쓴 것이 50줄로 가능하다면, 다시 쓴다.

**자문:** "시니어 엔지니어가 이걸 보고 '과도하게 복잡하다'고 할까?" 그렇다면 단순화한다.

---

## 7. 외과적 변경

**반드시 필요한 부분만 수정한다. 자기가 만든 잔해만 치운다.**

기존 코드를 편집할 때:
- 인접 코드, 주석, 포매팅을 "개선"하지 않는다.
- 고장 나지 않은 것을 리팩토링하지 않는다.
- 기존 스타일에 맞춘다, 자신이라면 다르게 했더라도.
- 관련 없는 죽은 코드를 발견하면 언급만 한다 — 삭제하지 않는다.

자기 변경으로 인해 고아가 된 경우:
- 자기 변경 때문에 쓸모없어진 import/변수/함수는 제거한다.
- 기존부터 있던 죽은 코드는 요청 없이 제거하지 않는다.

**검증 기준:** 변경된 모든 줄이 사용자의 요청에 직접 연결되어야 한다.

---

## 8. 목표 기반 실행

**성공 기준을 정의하고, 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 변환한다:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성하고, 통과시킨다"
- "버그 수정" → "재현하는 테스트를 작성하고, 통과시킨다"
- "X 리팩토링" → "리팩토링 전후로 테스트가 통과하는지 확인한다"

다단계 작업에는 간결한 계획을 명시한다:
```
1. [단계] → 검증: [확인 사항]
2. [단계] → 검증: [확인 사항]
3. [단계] → 검증: [확인 사항]
```

강한 성공 기준은 독립적으로 반복할 수 있게 해준다.
약한 기준("작동하게 해줘")은 끊임없는 명확화를 요구한다.

---

## 이 가이드가 제대로 작동하고 있다면:

- diff에 불필요한 변경이 줄어든다.
- 과도한 복잡성으로 인한 재작성이 줄어든다.
- 명확화 질문이 실수 이후가 아니라 구현 이전에 나온다.
